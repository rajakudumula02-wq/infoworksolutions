Requirements

Introduction: This feature delivers a multi-tenant, CMS-compliant HL7 FHIR R4 API platform designed as a sellable product for insurance companies (payers). The platform exposes healthcare data across twenty-one domains and supports CMS-9115-F (Patient Access API), CMS-0057-F (Payer-to-Payer), SMART App Launch Framework v2.0, and FHIR Bulk Data Access IG. Each payer tenant receives isolated data storage, tenant-specific configuration, and dedicated compliance reporting.

REQ-1 FHIR Server Foundation

User Story: As a platform engineer, I want a standards-compliant FHIR R4 server base, so that all APIs share consistent behavior, versioning, and metadata discovery.

Acceptance Criteria

  1. THE FHIR_Server SHALL expose a /metadata endpoint returning a valid HL7 FHIR R4 CapabilityStatement
  2. THE FHIR_Server SHALL support application/fhir+json as the default content type
  3. THE FHIR_Server SHALL support application/fhir+xml as an alternative content type via Accept header
  4. WHEN unsupported content type is sent, return HTTP 415 with OperationOutcome
  5. THE FHIR_Server SHALL include fhirVersion of 4.0.1 in all CapabilityStatement responses
  6. WHEN unknown endpoint is requested, return HTTP 404 with OperationOutcome
  7. THE FHIR_Server SHALL assign a unique logical id to every resource upon creation

REQ-2 Authentication and Authorization

User Story: As a security engineer, I want all API access controlled via SMART on FHIR, so that only authorized Clients can read or modify protected health information.

Acceptance Criteria

  1. Require SMART_Auth OAuth 2.0 bearer tokens for all resource endpoints
  2. Expired or invalid bearer token → HTTP 401 with OperationOutcome
  3. Resource outside granted scopes → HTTP 403 with OperationOutcome
  4. Support SMART scopes: patient/*.read, user/*.read, system/*.read
  5. Support PKCE in the SMART_Auth authorization code flow
  6. Access token expiry no more than 3600 seconds

REQ-3 Patient API

User Story: As a healthcare application developer, I want to retrieve and manage Patient resources in FHIR R4 format.

Acceptance Criteria

  1. Support read: GET /Patient/{id}
  2. Support search: GET /Patient with params _id, identifier, name, birthdate, gender, address-postalcode
  3. Support create: POST /Patient
  4. Support update: PUT /Patient/{id}
  5. Search returns FHIR Bundle of type searchset
  6. Validate against US Core Patient profile on create/update
  7. Profile validation failure → HTTP 422 with OperationOutcome
  8. Not found → HTTP 404 with OperationOutcome
  9. Include at least one identifier with system URI in every response

REQ-8 Payer-to-Payer API

User Story: As a payer organization, I want to exchange member clinical and claims data with other payers via FHIR for CMS interoperability compliance.

Acceptance Criteria

  1. Support $member-match operation on Patient resource
  2. Support Da Vinci PDex $everything operation

  4. Require mutual TLS (mTLS) in addition to SMART_Auth bearer tokens
  5. $everything returns Bundle with Patient, Coverage, EOB, and clinical resources
  6. Support _since parameter on $everything
  7. Unauthorized partner payer → HTTP 403 with OperationOutcome

REQ-9 Preauthorization API

  4. Support read: GET /ClaimResponse/{id}
  5. Update ClaimResponse disposition with approval/denial reason
  6. Support $inquire operation for status queries
  7. Missing clinical documentation → HTTP 422 with OperationOutcome
  8. Respond to $submit within 5 seconds for synchronous decisions

REQ-10 Pagination and Search Result Consistency

User Story: As an API consumer, I want consistent pagination across all search endpoints.

Acceptance Criteria

  1. Support _count parameter (default 20, max 100)
  2. Include next and self Bundle links when results exceed page size
  3. Support _sort parameter on all search interactions
  4. Support _include and _revinclude parameters
  5. Zero results → HTTP 200 with empty Bundle of type searchset

REQ-11 Audit Logging

User Story: As a compliance officer, I want all PHI access events logged for HIPAA audit trail requirements.

Acceptance Criteria

  1. Create AuditEvent for every read, search, create, update on Patient, EOB, Claim, ClaimResponse
  2. Record client identity, resource type, resource id, action type, and timestamp
  3. Retain AuditEvent records for minimum 6 years
  4. GET /AuditEvent returns events scoped to client's authorized patient population

REQ-12 Error Handling and Validation

User Story: As an API consumer, I want consistent, descriptive error responses.

Acceptance Criteria

  1. Return OperationOutcome for every 4xx and 5xx response
  2. Missing required field → HTTP 422 with issue.severity of error
  3. Internal error → HTTP 500 with issue.code of exception, no stack traces
  4. ResourceType mismatch → HTTP 400 with OperationOutcome

User Story: As a payer data analyst, I want to export large volumes of FHIR data in bulk using the FHIR Bulk Data Access IG.

Acceptance Criteria

  1. Support $export at system level: POST /$export
  2. Support $export at Patient level: POST /Patient/$export
  3. Support $export at Group level: POST /Group/{id}/$export
  4. Support _since parameter to filter by last updated date
  5. Support _type parameter to filter by resource type
  6. Support _outputFormat parameter (default: application/fhir+ndjson)
  7. Return all exported data in NDJSON format (one resource per line)
  8. Accepted → HTTP 202 with Content-Location header for polling

  10. Complete → HTTP 200 with JSON body containing transactionTime, request, requiresAccessToken, output array
  11. Require SMART_Backend_Services authentication (client credentials + signed JWT)

Design

Overview

The Healthcare FHIR API is an HL7 FHIR R4-compliant multi-tenant platform built on ASP.NET Core 8, exposing twenty-one clinical and administrative domains. The platform uses the Firely SDK (Hl7.Fhir.R4) for FHIR resource modeling and serialization, Entity Framework Core with SQL Server (JSONB-style storage) for persistence, Microsoft.AspNetCore.Authentication.JwtBearer for SMART on FHIR JWT validation, and Redis (StackExchange.Redis) for terminology caching.

Testing uses xUnit with FsCheck for property-based tests. The platform supports CMS-9115-F, CMS-0057-F, SMART App Launch Framework v2.0, and FHIR Bulk Data Access IG. Each payer tenant receives isolated data storage, tenant-specific configuration, rate limiting, and dedicated compliance reporting.

Middleware Pipeline

  1. TenantResolutionMiddleware — resolves tenant from subdomain, API key, or JWT claim
  2. FhirExceptionMiddleware — maps exceptions to OperationOutcome + HTTP status
  3. UseAuthentication — validates JWT bearer tokens
  4. UseAuthorization — enforces SMART on FHIR scope policies
  5. FhirContentNegotiationMiddleware — validates Content-Type / Accept headers
  6. RateLimitingMiddleware — enforces tenant-level rate limits
  7. AuditLoggingMiddleware — creates AuditEvent records for PHI interactions
  8. UseRouting / UseEndpoints — routes to controllers

Controller Pattern

All resource controllers follow the same pattern: inject repository, validation service, audit service, and pagination service. They return IActionResult with Firely Resource objects serialized by a custom FhirJsonOutputFormatter.

Pagination Engine

Default page size: 20. Maximum: 100. Bundle links include self, next, and previous as appropriate.

Terminology Service

Supported code systems: SNOMED_CT, LOINC, ICD-10, CPT, RxNorm. Results cached in Redis with 24-hour TTL.

Multi-Tenant Infrastructure

Tenant resolution strategies (in order): X-Api-Key header → JWT tenant_id claim → subdomain. All database queries scoped by TenantContext.TenantId. Rate limiting enforced per-tenant via sliding window counter in Redis.

Correctness Properties

Property 1: Content Negotiation

For any HTTP request, if Accept is application/fhir+json or absent → response is application/fhir+json; if application/fhir+xml → response is application/fhir+xml; otherwise → 415 with OperationOutcome.

Validates: Req 1.2, 1.3, 1.4

Property 2: Unknown Route Returns 404 OperationOutcome

For any unregistered path → 404 with valid OperationOutcome.

Validates: Req 1.6, 12.1

Property 3: Resource ID Uniqueness on Creation

For any N creation requests of the same type, all N returned id values shall be distinct.

Validates: Req 1.7

Property 4: Authentication Failure Returns 401

For any request with missing or invalid bearer token → 401 with OperationOutcome.

Validates: Req 2.1, 2.2

Property 5: Scope Violation Returns 403

For any request where token scopes don't include the required scope → 403 with OperationOutcome.

Validates: Req 2.3

Property 6: Access Token Expiry Bound

For any successful auth response, expires_in ≤ 3600.

Validates: Req 2.6

Property 7: Search Returns Bundle of Type Searchset

For any search interaction, response is a Bundle with type=searchset, including zero results.

Validates: Req 3.5, 10.5, 20.25, 23.10

Property 8: Profile Validation Rejects Invalid Resources with 422

For any create/update failing profile validation → 422 with OperationOutcome listing all errors.

Validates: Req 3.6, 3.7, 4.6, 4.7, 5.5–5.7, 6.6, 6.7, 7.4, 7.5, 13.6, 13.7, 15.6, 15.7, 16.6, 16.7, 18.8, 18.9, 19.6, 19.7, 20.3–20.24

Property 9: Not-Found Returns 404 OperationOutcome

For any read with non-existent id → 404 with OperationOutcome.

Validates: Req 3.8, 13.8, 15.8, 16.8, 19.8, 20.26, 23.11

Property 10: Patient Identifier with System URI

For any Patient response, at least one identifier with non-empty system URI.

Validates: Req 3.9
