📱 Member SMS Campaign

Requirements Document — v1.0

Introduction

The Member SMS Campaign application enables healthcare plan administrators to create, schedule, and send targeted SMS campaigns to members with active coverage. The application supports four campaign types: Welcome campaigns for new members, Referral campaigns to encourage member referrals, Encourage Utilization campaigns to promote use of covered benefits, and Holiday campaigns for seasonal greetings and health reminders.

In addition to automated campaigns, the system supports manual sending of both single SMS messages to individual members and bulk SMS messages to multiple members or filtered segments. The system manages its own member and coverage data in a SQL Server database and verifies member eligibility based on active coverage status before sending messages.

Glossary

TermDefinition
Campaign_ServiceThe core application service responsible for creating, scheduling, executing, and tracking SMS campaigns.
MemberA healthcare plan participant stored in the application's members table, managed via the Member API.
Active_CoverageA coverage record in the application database with a status of "active" and a period that includes the current date.
Welcome_CampaignA campaign type targeting newly enrolled members to introduce them to their plan benefits.
Referral_CampaignA campaign type targeting existing members to encourage them to refer others to the healthcare plan.
Utilization_CampaignA campaign type targeting members who have active coverage to encourage them to use their covered benefits.
Holiday_CampaignA campaign type targeting all members with active coverage to send holiday greetings or seasonal health reminders.
SMS_ProviderAn external third-party service (Azure Communication Services) used to deliver SMS messages.
Campaign_ScheduleA date and time specification that defines when the Campaign_Service sends campaign messages.
Eligibility_CheckThe process of verifying that a Member has Active_Coverage by querying the application's coverage table.
Campaign_RunA single execution instance of a campaign that tracks the delivery status of all messages sent.
Manual_Single_SMSA one-off SMS message sent to a specific Member, independent of any campaign.
Manual_Bulk_SMSAn immediate SMS send targeting multiple Members specified by a list of IDs or a filtered segment.

REQ-1 Create Campaign

As a plan administrator, I want to create an SMS campaign with a specific type and message content, so that I can prepare targeted outreach to members.

Acceptance Criteria

  1. WHEN a plan administrator submits a campaign creation request with a name, campaign type, and message template, THE Campaign_Service SHALL create a new campaign record with a status of "draft".
  2. THE Campaign_Service SHALL support exactly four campaign types: Welcome, Referral, Utilization, and Holiday.
  3. WHEN a campaign creation request is missing a required field (name, type, or message template), THE Campaign_Service SHALL return a validation error specifying the missing field.
  4. THE Campaign_Service SHALL enforce a maximum message template length of 160 characters.
  5. IF a campaign creation request contains an unsupported campaign type, THEN THE Campaign_Service SHALL return a validation error indicating the allowed types.

REQ-2 Schedule Campaign

As a plan administrator, I want to schedule a campaign for a future date and time, so that messages are sent at the optimal time for member engagement.

Acceptance Criteria

  1. WHEN a valid future date and time is provided for a draft campaign, THE Campaign_Service SHALL update the status to "scheduled" and store the schedule.
  2. IF a past date is provided, THEN THE Campaign_Service SHALL return a validation error.
  3. WHEN an already scheduled campaign is rescheduled, THE Campaign_Service SHALL replace the existing schedule.
  4. IF a campaign is not in "draft" or "scheduled" status, THEN scheduling SHALL be rejected.

REQ-3 Member Eligibility Verification

As a plan administrator, I want the system to verify member eligibility before sending messages, so that only members with active coverage receive SMS.

Acceptance Criteria

  1. WHEN a Campaign_Run begins, THE Campaign_Service SHALL query the coverage table for each targeted Member.
  2. A Member is eligible only when they have at least one coverage record with status "active" and a period that includes the current date.
  3. Members without Active_Coverage SHALL be excluded with reason "no active coverage".
  4. IF the database is unreachable, THE Campaign_Service SHALL retry up to 3 times with exponential backoff.

REQ-4 Campaign Execution

As a plan administrator, I want scheduled campaigns to execute automatically, so that members receive timely SMS without manual intervention.

Acceptance Criteria

  1. WHEN the scheduled time arrives, THE Campaign_Service SHALL initiate a Campaign_Run.
  2. Campaign status SHALL transition to "running" when the run begins.
  3. THE Campaign_Service SHALL send the message to each eligible Member via the SMS_Provider.
  4. WHEN complete, status SHALL transition to "completed" with a completion timestamp.
  5. Members without a phone number SHALL be skipped with reason "no phone number on file".

REQ-5 SMS Delivery

As a plan administrator, I want SMS messages delivered reliably, so that campaign outreach reaches the intended audience.

Acceptance Criteria

  1. Successful sends SHALL be recorded as "sent".
  2. Failed sends SHALL be recorded as "failed" with the failure reason.
  3. IF the SMS_Provider is unreachable, retry up to 3 times with exponential backoff.
  4. THE Campaign_Service SHALL track total sent, failed, and skipped counts per Campaign_Run.

REQ-6 Campaign Type Targeting Rules

As a plan administrator, I want each campaign type to target the appropriate member segment.

Acceptance Criteria

  1. Welcome: Members whose coverage started within the last 30 days.
  2. Referral: Members whose coverage started more than 30 days ago with active coverage.
  3. Utilization: All members with active coverage.
  4. Targeting rules SHALL be applied before eligibility checks.
  5. Holiday: All members with active coverage (same as Utilization).

REQ-7 Campaign Management

As a plan administrator, I want to view and manage my campaigns.

Acceptance Criteria

  1. List all campaigns with status, type, and schedule.
  2. Get campaign details including message template, schedule, status, and run history.
  3. Cancel a scheduled campaign — status transitions to "cancelled".
  4. Running or completed campaigns cannot be cancelled (return error).

REQ-8 Campaign Run Reporting

As a plan administrator, I want to view delivery reports for each campaign run.

Acceptance Criteria

  1. Report SHALL include total eligible, sent, failed, and skipped counts.
  2. Per-member delivery status breakdown with member ID, status, and reason.
  3. Multiple runs listed in reverse chronological order.

REQ-9 Manual Single SMS

As a plan administrator, I want to send a one-off SMS to a specific member without creating a campaign.

Acceptance Criteria

  1. Send SMS to a specified Member via the SMS_Provider.
  2. Enforce maximum message length of 160 characters.
  3. Perform Eligibility_Check before sending.
  4. Reject if member has no active coverage.
  5. Reject if member has no phone number on file.
  6. Record the send event with member ID, message, delivery status, and timestamp.
  7. Record delivery failures with the failure reason.

REQ-10 Manual Bulk SMS

As a plan administrator, I want to send bulk SMS to multiple members immediately without a scheduled campaign.

Acceptance Criteria

  1. Accept a list of member IDs and a custom message.
  2. Accept a Member_Segment filter (e.g., all active coverage) and resolve to member list.
  3. Enforce maximum message length of 160 characters.
  4. Perform Eligibility_Check for each member — send only to eligible members.
  5. Exclude ineligible members with reason "no active coverage".
  6. Skip members without phone numbers with reason "no phone number on file".
  7. Send immediately — do not store as a scheduled campaign.
  8. Return summary report: total sent, failed, and skipped with reasons.
  9. Continue sending on individual failures — do not halt the batch.

REQ-11 Holiday Campaign Scheduling Restriction

As a plan administrator, I want Holiday campaigns to only be scheduled for November or December.

Acceptance Criteria

  1. Accept scheduling for dates in November (month 11) or December (month 12).
  2. Reject scheduling for dates outside November/December with a validation error.
  3. Validate the new date on reschedule as well.

REQ-12 Member Management API

As a plan administrator, I want to manage member records via API.

Acceptance Criteria

  1. Create member with first name, last name, date of birth, and phone number — return member ID.
  2. Update member's phone number, name, and date of birth via PUT.
  3. Get member by ID with full record including associated coverage records.
  4. List all members with pagination (default 50, max 100).
  5. Return validation error if required fields (first name, last name) are missing.
  6. Search members by name or phone number.

REQ-13 Coverage Management API

As a plan administrator, I want to manage coverage records via API.

Acceptance Criteria

  1. Create coverage with member ID, plan name, status, effective date, and expiration date.
  2. Update coverage status and period dates via PUT.
  3. Get all coverages for a specific member.
  4. List all active coverages with pagination.
  5. Return validation error if member ID does not exist.
  6. Support bulk import of member and coverage records via CSV upload.
